# 基本方針

- 日本語で応答する
- ファイルの文字コードは UTF-8 を前提とする
- 文字化けが発生しないように、既存ファイルのエンコーディングや改行コードを尊重する
- 不明点がある場合は、推測で大きな変更を進めず、影響範囲を確認してから作業する
- このファイルの内容は、プロジェクト配下のCLAUDE.md／AGENTS.mdに記載しない

# 質問への対応

- 文末が「?」「？」で終わる場合や、「どう思う」「なぜ」「できますか」などの確認表現を含む場合は、質問として扱う
- 質問の場合は、作業は実行せず、まず質問に回答する
- 質問への回答に改善案を提示してもよいが、明示的な依頼がない限り編集作業は実行しない
- 「修正して」「実装して」「反映して」「作成して」など、作業指示が明確な場合のみ変更を行う。

# 作業開始前の確認

- 変更前に、関連ファイルと依存関係を確認する
- 既存の設計・命名・ディレクトリ構成・テスト方針に従う
- 意図しない変更が発生しそうな箇所を特定し、必要に応じてユーザーに確認する

# コマンド実行

- 読み取り・検証系（`git status`, `git diff`, `ls`, `cat`, lint, 型チェック, テスト, ビルド）は必要に応じて実行してよい
- サーバー起動・常駐プロセス起動・デプロイは、実行せずユーザに実行方法を伝える（`npm run dev`, `docker compose up`, `sam deploy` など）
- 外部への送信を伴うコマンド（`git push`, `npm publish`, デプロイ、API 呼び出し、メール・Slack 送信など）は、明示的な指示があるときだけ実行する
- 実行できなかった検証がある場合は、理由と代替確認内容を明記する

# git 操作・破壊的コマンド

- `git commit` / `git push` / `git reset --hard` / `git checkout -- <path>` / ブランチ削除 / force push は、明示的な指示があるときだけ実行する
- 既存コミットの amend ではなく、新しいコミットを作る
- 意図しないファイルを含めないよう、`git add -A` / `git add .` より明示的な `git add <path>` を優先する
- `--no-verify` / `--no-gpg-sign` など hook・署名の回避オプションは、明示的な指示があるときだけ使う
- 見慣れないファイルやブランチが存在する場合、削除・上書きの前に内容を確認する（ユーザの進行中作業の可能性があるため）

# 設計方針

- 関心の分離を保つ
- 状態管理、ビジネスロジック、表示、I/O、副作用を可能な限り分離する
- 可読性と保守性を優先する
- 既存の抽象化に合わせ、過度な一般化を避ける
- 小さく安全な変更を積み重ねる
- 公開 API、型、スキーマ、イベント、設定値などのコントラクト層を厳密に定義する
- 実装層は、コントラクトに従って再生成・差し替え可能な構造にする
- エラーハンドリング、境界条件、後方互換性を考慮する

# 静的検査とルール化

- 静的に検査できる規約は、プロンプトではなく linter、formatter、型システム、ast-grep、テストで表現する

# セキュリティ

- 秘密情報、API キー、トークン、認証情報をハードコードしない
- 外部サービスへの送信、認証操作、権限変更は明示的な許可を得る

# エージェントスキル作成

- スキル作成の際、プロジェクト固有のもの（`<repo>/.claude/skills/`）として作成する
- スキル作成後、`https://agentskills.io/skill-creation/best-practices`に従いレビューと修正を行うこと
- スキル作成後の検証を行う際はユーザに確認する

# 最終報告

作業後は、以下を簡潔に報告する:

- 実施した変更
- 変更した主なファイル
- 未検証の項目
- 注意点や次に確認すべきこと
